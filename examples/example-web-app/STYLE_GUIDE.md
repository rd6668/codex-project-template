# Style Guide

## General

- prefer the smallest maintainable change
- avoid unrelated edits
- keep code easy to read before optimizing abstractions

## Language

- use one formatter and one linter consistently
- prefer descriptive names over shortened names
- keep files aligned with existing project conventions

## API

- keep endpoints resource-oriented
- use UTC for server timestamps
- return predictable error shapes

## Database

- use explicit migrations for schema changes
- keep naming consistent across tables and fields

## Testing

- add tests for new behavior
- add regression tests for bug fixes

## Documentation

- update architecture, API, and decisions after major changes
